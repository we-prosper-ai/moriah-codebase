## Support Categories & Playbooks

### Category 1: Technical Issues

**Symptoms:**
- "App won't load"
- "CSV upload failed"
- "Can't log in"
- "Error message: [X]"

**First Response (Automated by Moriah):**
```
Hi [Name],

Thanks for reaching out. I'm looking into this right now.

Quick diagnostic questions:
1. What browser are you using?
2. Did you get an error message? If so, what does it say?
3. Did this work before, or is this your first time?

Sending you a fix in the next 30 minutes. This is exactly the kind of early feedback that makes us better.

— Tina
```

**Resolution Playbook:**
1. Check browser console (JS errors)
2. Check server logs (backend errors)
3. Reproduce the issue (can we see it?)
4. Send specific fix or workaround
5. Follow up 1 hour later ("Is it working now?")

**Success Metric:** Tech issues resolved in <24 hours, 100% of them.

---

### Category 2: Feature Questions

**Symptoms:**
- "How do I [do thing X]?"
- "Is it possible to [feature]?"
- "Where's the [button/page]?"

**First Response (Automated by Moriah):**
```
Hi [Name],

Great question. Here's how [feature] works:

[Short explanation]

If that doesn't answer it, just reply and we'll dig deeper. I actually love these questions because they tell me what's confusing.

— Tina
```

**Resolution Playbook:**
1. Provide clear answer (with screenshots if complex)
2. Offer alternative if asking for non-existent feature
3. Log for future documentation
4. Ask: "Did that help?"

**Success Metric:** Feature questions answered in <1 hour, documented for FAQ.

---

### Category 3: Feature Requests

**Symptoms:**
- "It would be cool if [X]"
- "Can you add [feature]?"
- "I wish it did [thing]"

**First Response (Moriah):**
```
Hi [Name],

Love this idea. We're actively building Phase 2 features based on beta feedback, and this is the exact kind of request that shapes what comes next.

Quick question: How would you use [feature] in your actual financial life?

We prioritize features by impact + frequency of request. If other users want this too, it moves up the list.

Thanks for helping us build something useful.

— Tina
```

**Resolution Playbook:**
1. Acknowledge idea (don't dismiss)
2. Ask clarifying questions (understand the actual problem)
3. Evaluate: Does this fit our roadmap?
4. Communicate: "This is roadmap for Phase 2" or "This isn't our direction, here's why"
5. Log for roadmap review

**Success Metric:** All feature requests acknowledged within 2 hours.

---

### Category 4: Pricing/Business Questions

**Symptoms:**
- "Why does it cost?"
- "Is there an annual plan?"
- "How is this different from [competitor]?"

**First Response (Moriah):**
```
Hi [Name],

Good questions. Here's how we think about pricing:

[Pricing rationale]

We're in beta through March 31, so access is free. After that, we'll have [pricing plan].

This is exactly the conversation we want to have before we finalize pricing. What would feel fair to you?

— Tina
```

**Resolution Playbook:**
1. Explain pricing philosophy (not defensively, just honestly)
2. Offer early-user pricing if applicable
3. Ask what would feel fair (gather data)
4. Log for pricing review meeting

**Success Metric:** Pricing conversations feel transparent, not pushy.

### Category 5: Data/Security Questions

**Symptoms:**
- "Is my data safe?"
- "Who has access to my [data]?"
- "Can I delete my account?"
- "Where's the privacy policy?"

**First Response (Moriah, immediate):**
```
Hi [Name],

Important question. Here's how we handle data:

- Your transactions stay in YOUR account (encrypted)
- We never sell data (ever)
- You can delete anytime
- Privacy policy: [LINK]
- Security: [Technical details]

You should trust that your financial data is private. If you have other concerns, reply and we'll address them directly.

— Tina
```

**Resolution Playbook:**
1. Answer completely (don't deflect)
2. Provide proof (privacy policy, security details)
3. Offer to discuss on call if complex
4. Document the question (might need FAQ expansion)

**Success Metric:** Security questions answered in <30 minutes with complete transparency.

### Category 6: Feedback (Unsolicited)

**Symptoms:**
- "Here's what I like: [X]"
- "Here's what I don't like: [Y]"
- "My experience using this was: [narrative]"

**First Response (Moriah):**
```
Hi [Name],

This feedback is gold. Honest reactions from real users is how we get better.

I'm forwarding this to my team right now. We review all feedback weekly and use it to shape what we build next.

Thank you for using it and for telling us what you actually think.

— Tina
```

**Resolution Playbook:**
1. Acknowledge immediately (shows you're listening)
2. Forward to Tina (weekly feedback review)
3. Extract data (what specifically was good/bad?)
4. Follow-up (let them know what changed because of their feedback)

**Success Metric:** Every piece of feedback logged, reviewed weekly with Tina.

## Escalation Playbook

**When to involve Tina immediately:**
- Security/privacy concern
- User very upset/angry
- Feature request that contradicts strategy
- Legal/compliance question
- Data loss or corruption

**How to escalate:**
1. Don't try to resolve yourself
2. Acknowledge to user: "This needs special attention, getting expert now"
3. Message Tina directly: "Flag: [category] from [user], needs your input"
4. Await Tina's response before replying to user

## Support Tone (Tina's Voice)

✅ **Do this:**
- Be honest (even if the answer is "we don't do that yet")
- Be fast (people's time is valuable)
- Be specific (not generic platitudes)
- Ask clarifying questions (understand actual problem)
- Follow up ("Did that solve it?")

❌ **Don't do this:**
- Deflect with "sorry for the inconvenience"
- Generic customer service scripts
- Assume you know the problem
- Disappear after first response
- Treat them like they're being needy
